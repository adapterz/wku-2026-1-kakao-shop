## [2026-07-09] product.html 내용을 order.html에 잘못 붙여넣음
- 증상: git diff 확인 시 product.html의 헤더 주석과 내용이 order.html 것으로 통째로 바뀌어 있었음
- 원인: 두 파일을 번갈아 작업하다 코드를 잘못된 파일에 붙여넣음
- 해결: git restore public/product.html로 마지막 커밋 상태로 되돌린 후 재작업
- 배운 점: 커밋 안 된 파일은 git restore로 간단히 되돌릴 수 있음. 여러 파일 동시 작업 시 붙여넣기 전 파일명 재확인 필요

## [2026-07-09] 브랜치를 안 바꾸고 이전 브랜치에 계속 작업함
- 증상: 여러 화면(product, giftbox 등) 작업 후 push 하려는데 이전 작업 브랜치 그대로였음
- 원인: 새 화면 작업 시작할 때 git checkout -b로 새 브랜치 만드는 것을 매번 깜빡함
- 해결: 커밋 전 상태였기 때문에 git checkout -b <새브랜치>로 이동 시 수정 내용이 그대로 따라와 문제없이 해결
- 배운 점: 커밋하기 전에는 브랜치를 새로 만들어도 작업 내용이 유지된다는 것을 확인함. 화면 작업 시작 전 git branch로 항상 확인하는 습관 필요

## [2026-07-10] git add 후 commit 시 "nothing to commit" 발생
- 증상: git add public/index.html public/js/home.js public/js/components.js 실행 후 커밋하려는데 "nothing to commit, working tree clean" 메시지 출력
- 원인: 새 브랜치(feature/fe-home-fix) 생성 이전에 이미 다른 시점에서 동일 내용을 커밋해버린 상태였음
- 해결: git log --oneline으로 최근 커밋 목록을 확인해 이미 반영되어 있음을 확인
- 배운 점: add 이후 commit이 안 될 때는 파일 저장 여부뿐 아니라, 혹시 이미 커밋된 상태는 아닌지 git log로 먼저 확인하는 습관이 필요함

## [2026-07-13] 히어로 헤드라인이 "이동해 / 요"처럼 글자 중간에서 줄바꿈됨
- 증상: h2에 <br>로 줄을 2줄("마룡이와 함께" / "익산을 편하게 이동해요")로 나눴는데도, 실제 화면에는 2번째 줄이 다시 "이동해"와 "요"로 쪼개져 총 3줄로 보임
- 원인: .hero-text의 max-width가 64%(약 229px)로 좁게 잡혀 있어서, 25px 폰트 기준 "익산을 편하게 이동해요" 텍스트 실제 폭(약 263px)이 컨테이너보다 넓어 브라우저가 자동으로 다시 줄바꿈함
- 해결: 브라우저에서 임시 span으로 실제 텍스트 폭을 직접 측정(폰트 크기별 263px→220px 등)해 필요한 최소 폭을 확인한 뒤, 폰트를 22px로 줄이고 max-width를 68%로 늘리고 h2에 white-space: nowrap을 추가. 이후 Range.getClientRects()로 실제 글자 우측 끝(247px)과 마스코트 좌측 끝(254px) 사이 여백(7px)을 재확인해 겹치지 않는 것까지 검증
- 배운 점: 여러 줄 텍스트에 <br>을 넣어도, 각 줄 자체가 컨테이너보다 넓으면 브라우저가 또 줄바꿈한다. "몇 줄로 보일지"는 <br> 개수가 아니라 각 줄의 실제 렌더링 폭과 컨테이너 폭을 같이 봐야 함. 눈대중 대신 임시 span/Range로 실제 렌더링 폭을 재보면 정확하게 진단 가능

## [2026-07-13] 석탑 장식이 마스코트와 겹치는 것처럼 보였지만 실제로는 안 겹침
- 증상: 히어로 배너에 미륵사지 석탑 실루엣을 추가하는데, getBoundingClientRect() 기준으로는 석탑 장식 박스와 마스코트 이미지 박스가 계속 겹쳤음(overlapsMascotBox: true)
- 원인: 마스코트 이미지가 투명 배경 PNG라서, 사각형 바운딩 박스는 겹쳐도 실제 캐릭터 그림이 그려진 픽셀은 그 영역에 없을 수 있음. 바운딩 박스만으로는 "진짜 겹쳐 보이는지" 알 수 없었음
- 해결: <canvas>에 마스코트 이미지를 그린 뒤, 겹치는 좌표 영역만 getImageData()로 알파 채널을 읽어서 실제 불투명 픽셀 비율을 계산. 처음엔 9%가 겹쳐서 위치를 더 조정했고, 최종적으로 0%(전혀 안 겹침)를 확인한 뒤에 배치 확정
- 배운 점: 투명 PNG/SVG가 섞인 레이아웃에서는 바운딩 박스 겹침 여부가 실제 시각적 겹침과 다를 수 있다. 픽셀 단위로 확실히 확인하고 싶으면 canvas + getImageData로 알파값을 직접 검사하면 정확하게 판단할 수 있음

## [2026-07-13] 픽셀은 안 겹쳐도 "실제로 잘 안 보이는" 디자인이 될 수 있음
- 증상: 석탑 실루엣이 마스코트/텍스트와 픽셀 단위로는 안 겹치는 걸 검증까지 했는데도, 사용자가 보기엔 "안 어울린다"고 피드백함
- 원인: 겹침 검사는 "깨졌는지"만 확인하는 것이고, "장식이 실제로 눈에 띄고 잘 어울리는지"는 별개 문제였음. 마스코트 머리 위 26px 짜리 좁은 공간에 작게 넣다 보니, 마스코트 바운딩 박스와는 안 겹쳐도 시각적으로 존재감이 없고 위치도 어색했음
- 해결: 세로로 긴 탑 모양 대신 낮고 넓은 실루엣으로 다시 그리고, 마스코트와 아예 다른 위치(좌하단)의 넉넉한 공간에 5배 가까이 크게 배치. 필요하면 히어로 자체의 여백(padding-bottom)을 늘려서 공간을 만듦
- 배운 점: "안 겹침"은 최소 조건이지 충분 조건이 아니다. 장식 요소를 넣을 땐 겹침 여부뿐 아니라 "그 크기와 위치에 실제로 의도가 드러나는지"까지 확인해야 한다. 좁은 여백에 욱여넣기보다, 필요하면 레이아웃(패딩 등) 자체를 조정해 정당한 공간을 만드는 게 나을 때가 있음

## [2026-07-14] USE_DUMMY_PRODUCTS=true로 설정해도 로컬에서 계속 500 에러
- 증상: .env에 USE_DUMMY_PRODUCTS=true를 넣고 npm run dev로 새로 띄웠는데도 GET /api/products가 계속 internal_server_error(500)를 반환함. 서버를 몇 번을 다시 켜도 재현됨
- 원인: nodemon은 자식 node 프로세스가 죽으면 "감시"하고 있다가 자동으로 재시작하는 도구임. taskkill로 node.exe(자식)만 죽이면, 그 부모인 nodemon이 곧바로 새 자식을 다시 띄워버려서 프로세스가 실제로는 안 꺼짐. 이 과정이 여러 세션에 걸쳐 반복되면서 각기 다른 시점의 .env를 읽은 여러 서버 프로세스가 같은 3000번 포트를 두고 계속 남아있게 됨 (그 중 하나만 포트를 실제로 점유하는데, 그게 어떤 시점의 프로세스인지 알 수 없어 원인 파악이 어려움)
- 해결: `netstat -ano`로 3000번 포트를 실제 점유 중인 PID를 확인 → `wmic process where "commandline like '%server/app.js%'" get ProcessId,ParentProcessId,CommandLine`로 nodemon/cmd까지 이어지는 전체 프로세스 트리를 확인 → 자식(node)이 아니라 최상위 cmd.exe를 `taskkill /F /T /PID <pid>`(트리 전체 종료)로 꺼야 확실하게 없어짐
- 배운 점: nodemon처럼 "자동 재시작"하는 프로세스 감시 도구가 떠 있을 때는 자식 프로세스 하나만 죽이면 안 된다. 완전히 끄려면 최상위 프로세스를 트리째(/T) 종료해야 하고, 서버가 이상하게 동작할 땐 새로 켜기 전에 항상 포트 점유 프로세스와 관련 프로세스 트리를 먼저 확인하는 게 안전함

## [2026-07-14] PowerShell 실행 정책 제한으로 인한 npm 실행 실패 및 nodemon 의존성 에러
- 증상: 터미널에서 npm run dev 실행 시, PowerShell 보안 정책(PSSecurityException) 및 UnauthorizedAccess 오류가 발생하며 스크립트가 실행되지 않음
- 원인: 윈도우 기본 PowerShell 정책상 npm.ps1 등의 외부 스크립트 실행이 차단되어 있음
- 해결: npm.cmd run dev 와 같이 윈도우 커맨드 스크립트(.cmd)를 명시적으로 호출해 실행 정책 제약을 우회함. 이후 nodemon 부재로 인한 에러는 npm.cmd start(node server/app.js)를 사용하여 nodemon 의존성 없이 서버를 구동해 해결함
- 배운 점: Windows PowerShell 환경에서는 실행 권한 문제로 CLI 스크립트(.ps1) 실행이 제한될 수 있으므로, .cmd 버전을 직접 호출하거나 실행 정책을 적절히 설정해야 함

## [2026-07-15] 다크모드에서 페이지 이동 시 라이트모드가 잠깐 표시됨
- 증상: 다크모드를 사용 중인데 다른 HTML 페이지로 이동하면 기본 라이트 화면이 먼저 보이고 잠시 후 다크모드로 변경되는 현상이 간헐적으로 발생함
- 원인: 공통 `api.js`와 일부 페이지 JavaScript가 `DOMContentLoaded` 이후 `<body>`에 `dark-theme` 클래스를 추가함. 브라우저가 기본 CSS로 첫 화면을 그린 다음 테마를 변경해 FOUC(스타일 전환 플래시)가 발생함
- 해결: 모든 HTML의 `<head>`에서 CSS보다 먼저 `public/js/bootstrap.js`를 동기 실행하고, 저장된 테마를 `<html>` 요소에 즉시 적용함. `api.js`, `profile.js`, `settings.js`의 중복·지연 초기화 코드를 제거하고 설정 토글도 `document.documentElement`를 기준으로 통일함
- 검증: 전체 HTML에서 `bootstrap.js`가 `style.css`보다 먼저 선언되어 있는지 확인하고, 다크모드 저장 상태에서 페이지를 이동해 첫 화면부터 다크 테마가 적용되는지 확인함
- 배운 점: 첫 화면의 색상처럼 렌더링 전에 결정되어야 하는 상태는 `DOMContentLoaded` 이후에 적용하면 늦다. 공통 부트스트랩을 CSS보다 먼저 실행하고 테마 초기화 책임을 한 곳에 모아야 신규 페이지에서도 같은 문제가 반복되지 않음

## [2026-07-15] 선물함과 프로필을 열 때마다 동일한 DB 조회를 기다림
- 증상: 선물함과 프로필을 오갈 때마다 미사용·사용완료 패스 로딩 문구가 표시되고, DB 응답이 도착할 때까지 목록과 이용 현황이 늦게 나타남
- 원인: 두 페이지가 진입할 때마다 `/api/gifts?status=unused`, `/api/gifts?status=used`를 각각 호출함. 같은 로그인 세션에서 방금 조회한 데이터가 있어도 재사용하지 않아 동일한 API·DB 조회가 반복됨
- 검토: 선물 목록을 쿠키에 저장하면 크기 제한이 있고 모든 HTTP 요청에 자동 첨부되어 불필요한 네트워크 비용과 개인 데이터 노출 위험이 생김
- 해결: 공통 `api.js`에 5분 TTL의 `sessionStorage` 캐시를 추가하고 선물함과 프로필이 `fetchGiftCollections()`를 함께 사용하도록 통합함. 주문 생성·패스 사용·로그인·로그아웃 성공 및 `login_required` 응답에서는 캐시를 즉시 삭제함
- 검증: 같은 탭에서 `fetchGiftCollections()`를 두 번 실행했을 때 첫 번째 호출에서만 API 2건이 발생하고 두 번째 호출은 저장값을 반환하는지 확인함. 캐시 삭제 후에는 API가 다시 호출되는 것도 확인함
- 배운 점: 자주 이동하는 화면의 동일 조회는 짧은 수명의 브라우저 캐시로 체감 속도를 개선할 수 있다. 다만 캐시는 서버 인증을 대체하지 않으며, 데이터 변경 지점마다 무효화하지 않으면 오래된 상태를 보여줄 수 있으므로 TTL과 명시적 삭제를 함께 설계해야 함
