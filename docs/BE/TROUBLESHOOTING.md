# BE TROUBLESHOOTING

## [2026-07-08] MySQL 미설치로 로컬 DB 테스트 불가
- 증상: PowerShell에서 `mysql -u root -p` 실행 시 명령어를 찾을 수 없다는 오류 발생
- 원인: 로컬 PC에 MySQL CLI가 설치되어 있지 않거나 PATH 등록이 되어 있지 않았음
- 해결: 로컬 테스트와 EC2 DB 테스트 범위를 분리하고, EC2 DB 세팅 후 API 응답으로 DB 연동 여부를 확인
- 배운 점: SQL 파일만 있어서는 DB 테스트가 되지 않으며, 로컬이든 EC2든 실제 MySQL 실행 환경이 필요함

## [2026-07-08] 더미 데이터와 DB 데이터 응답 출처 혼동
- 증상: API 응답과 MySQL SELECT 결과가 비슷하게 보여 더미 데이터인지 DB 데이터인지 구분이 어려웠음
- 원인: DB 연결 전 테스트를 위해 product 라우터에 더미 데이터를 넣어 둔 상태였고, 이후 EC2 DB에도 유사한 상품 데이터가 들어감
- 해결: 상품 조회 라우터에서 더미 데이터 분기를 제거하고 실제 DB 조회 기준으로 정리
- 배운 점: M1 이후에는 코드 더미 데이터보다 DB seed/insert 데이터를 기준으로 응답 출처를 명확히 해야 함

## [2026-07-09] API 명세 경로와 실제 auth 라우터 경로 불일치
- 증상: API 명세서에는 회원 기능이 `/api/users/...` 기준으로 보였지만, 실제 코드는 `/api/auth/...` 기준으로 구현됨
- 원인: 인증 기능을 별도 auth 라우터로 분리하면서 명세서 경로와 코드 경로가 달라짐
- 해결: 이미 FE/BE가 `/api/auth/...` 기준으로 맞춰져 있어 코드 대신 명세서 경로를 수정하는 방향으로 정리
- 배운 점: 기능 구현 후에는 API 명세서와 실제 라우터 경로를 함께 점검해야 함

## [2026-07-09] 304 Not Modified 응답을 API 오류로 오해
- 증상: Chrome DevTools Network 탭에서 `/api/products` 응답이 200이 아니라 304로 표시됨
- 원인: 브라우저 캐시 검증 과정에서 서버가 이전 응답을 그대로 사용해도 된다고 알려준 정상 응답이었음
- 해결: 화면에 상품 목록이 정상 표시되는지 확인하고, 필요 시 DevTools의 Disable cache 또는 Ctrl+F5로 200 응답 재확인
- 배운 점: 304는 API 실패가 아니며, 개발 중에는 캐시 여부를 함께 확인해야 함

## [2026-07-10] 상품 조회 응답에 불필요한 createdAt 노출
- 증상: 상품 목록/상세 API 응답에 FE에서 사용하지 않는 `createdAt`이 포함되어 있었음
- 원인: DB 컬럼을 API 응답으로 그대로 매핑하면서 내부 관리용 필드가 프론트 응답에 포함됨
- 해결: 상품 조회 응답에서는 `createdAt`을 제외하고, DB의 `created_at`은 내부 관리용으로 유지
- 배운 점: DB 컬럼과 API 응답 필드는 항상 같을 필요가 없으며, FE에 필요한 값만 응답하는 것이 좋음

## [2026-07-10] `/api/gifts`에 사용자 id가 없어서 선물함 조회 기준 혼동
- 증상: 기존 설계처럼 `/api/users/{id}/gifts` 형태가 아니라 `/api/gifts`만 있어도 특정 사용자의 선물함을 조회할 수 있는지 헷갈렸음
- 원인: URL의 userId가 아니라 로그인 시 세션에 저장한 `req.session.user.id`를 기준으로 DB를 조회하는 구조였음
- 해결: 로그인 API에서 DB user id를 세션에 저장하고, 선물함 API에서는 세션 user id로 `gifts` 테이블을 조회하는 흐름으로 이해
- 배운 점: 내 선물함처럼 로그인 사용자 기준의 API는 URL에 userId를 직접 노출하기보다 세션 기준으로 처리하는 편이 자연스럽고 안전함

## [2026-07-13] 회원가입 생년월일 연도 과입력 허용
- 증상: 회원가입 화면에서 `2000022-08-11`처럼 4자리를 초과한 연도 입력이 가능한 경우가 있었음
- 원인: 브라우저의 `type="date"` 입력 UI만 믿고, FE에서 생년월일 형식 검증을 따로 하지 않았음
- 해결: FE에서 `YYYY-MM-DD` 형식, 1900년 이후, 미래 날짜 불가 조건을 검증하고 BE에서도 같은 기준으로 최종 차단
- 배운 점: HTML 입력 타입은 사용자 편의 장치이고, 실제 데이터 규칙은 FE와 BE에서 명확히 검증해야 함

## [2026-07-13] 선물 사용 완료 안내 문구가 어색하게 보임
- 증상: 사용 완료 처리 후 기본 팝업의 제목/문구가 사용자 입장에서 어색하게 보일 수 있었음
- 원인: 브라우저 기본 `alert`는 팝업 제목을 직접 제어하기 어렵고, 기존 안내 문구도 처리 중심 표현이었음
- 해결: 완료 안내 문구를 `패스 사용이 완료되었습니다.`로 수정
- 배운 점: 기본 팝업은 제목 제어가 제한되므로, 필요한 경우 이후 커스텀 모달로 분리하는 것이 좋음

## [2026-07-14] 패스 상위 카테고리 필터 결과가 0건으로 표시됨
- 증상: 기본 이동·환승/광역·관광/특수 버튼을 선택하면 실제 상품이 있는데도 결과가 0건으로 표시됨
- 원인: 화면의 상위 분류 코드(`basic`, `transfer`, `special`)와 DB의 세부 카테고리 값(`daily_pass`, `monthly_pass` 등)을 직접 비교함
- 해결: BE에 상위 분류와 세부 카테고리의 매핑을 정의하고, 허용된 분류만 검증한 뒤 `category IN (?, ...)` 조건으로 조회
- 배운 점: 화면용 분류와 DB 저장값이 다르면 변환 책임을 한 곳에 두고, 임의 문자열이 아닌 허용 목록으로 관리해야 함

## [2026-07-14] 비회원 친구 선물을 기존 회원 중심 DB 구조에 저장할 수 없음
- 증상: 가입하지 않은 번호로 선물하면 연결할 회원 ID가 없어 주문과 선물 발급을 저장할 수 없음
- 원인: `orders.receiver_id`, `gifts.receiver_id`가 반드시 회원을 참조하는 구조였고 주문 당시 수신자 이름·번호를 별도로 보관하지 않았음
- 해결: 두 `receiver_id` 컬럼에 `NULL`을 허용하고 `orders.receiver_name`, `orders.receiver_phone`에 입력값을 저장함. 번호가 활성 회원 한 명과 일치할 때만 회원 선물함에 연결하고 나머지는 SMS Mock으로 처리
- 배운 점: 공공 교통 티켓처럼 비회원도 받을 수 있는 도메인에서는 회원 연결 정보와 실제 전달 대상 정보를 분리해야 함

## [2026-07-15] 상품 검색어를 SQL에 직접 연결할 경우 SQL Injection과 와일드카드 오용 위험
- 증상: 검색 기능 구현 시 사용자 검색어를 `LIKE '%검색어%'` 문자열로 직접 조합하면 SQL 구문이 변조되거나 `%`, `_` 입력이 전체 상품과 일치할 수 있음
- 원인: 검색어를 SQL 문장의 일부로 취급하고 LIKE 전용 특수문자를 별도로 처리하지 않는 구현 방식
- 해결: `name LIKE ?` 파라미터 바인딩을 사용하고 `%`, `_`, 역슬래시를 이스케이프함. 검색어는 공백 제거 후 1~100자로 검증
- 검증: 정상 상품명은 검색되고, 없는 상품명·SQL Injection 형태·`%` 입력은 모두 0건이며 빈 검색어는 HTTP 400 `invalid_keyword`로 처리됨
- 배운 점: 파라미터 바인딩은 SQL Injection을 막고, LIKE 와일드카드 이스케이프는 검색 의미를 통제하므로 두 처리가 모두 필요함

## [2026-07-15] 상품 검색 결과가 없을 때 서버 오류로 오해할 수 있음
- 증상: 검색 결과가 0건일 때 데이터가 없는 정상 상황인지 API 오류인지 화면에서 구분하기 어려움
- 원인: 빈 목록에 대한 API·화면 처리 규칙이 명확하지 않았음
- 해결: BE는 HTTP 200과 `data: []`를 반환하고, FE는 `검색 결과가 없습니다` 안내 화면을 표시하도록 역할을 분리
- 배운 점: 빈 목록은 실패가 아닌 정상 응답으로 정의하고, 사용자 안내는 FE에서 명확히 처리하는 것이 좋음

## [2026-07-15] GitHub Actions에서 PEM SSH 개인키를 인식하지 못함
- 증상: GitHub Actions가 EC2에 접속할 때 `Load key "/home/runner/.ssh/id_rsa": error in libcrypto`와 `Permission denied (publickey)` 오류가 발생함
- 원인: 여러 줄로 구성된 PEM 개인키를 GitHub Secret에서 파일로 복원하는 과정에서 줄바꿈 형식이 유지되지 않아 Runner가 올바른 SSH 키로 인식하지 못함
- 해결: 로컬 PEM 파일을 Base64 한 줄 문자열로 변환해 `SSH_PRIVATE_KEY_B64` Secret으로 등록하고, 워크플로에서 `base64 --decode`로 개인키 파일을 복원함. `ssh-keygen -y` 명령으로 접속 전에 키 형식도 검증함
- 검증: SSH 설정 단계가 통과하고 GitHub Actions Runner에서 EC2 접속 및 원격 배포 명령 실행에 성공함
- 배운 점: 여러 줄 Secret은 실행 환경의 줄바꿈 차이로 손상될 수 있으므로 Base64로 전달하고 실행 시 복원하면 형식 문제를 줄일 수 있음

## [2026-07-15] Docker 컨테이너 시작 직후 헬스 체크 실패
- 증상: SSH 접속, main 코드 갱신, Docker 이미지 빌드와 컨테이너 실행은 성공했지만 `curl: (7) Failed to connect to 127.0.0.1 port 3001` 오류로 Actions 작업이 실패함
- 원인: `docker compose up -d`가 컨테이너 시작 요청만 완료한 직후 Express 서버가 준비되기 전에 헬스 체크를 실행함
- 해결: `docker compose up -d --wait --wait-timeout 90`을 사용해 Compose의 `healthy` 상태를 기다린 뒤 `/api/health`를 호출하도록 변경함
- 검증: Actions 배포 작업이 성공하고 Docker 컨테이너가 `running / healthy` 상태이며 운영 도메인의 서버·DB 헬스 체크도 정상 응답함
- 배운 점: 컨테이너 실행 완료와 애플리케이션 준비 완료는 다르므로 배포 자동화에서는 선언된 헬스 체크가 통과할 때까지 기다려야 함
